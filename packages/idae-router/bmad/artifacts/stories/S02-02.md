## User Story

As a **router user**, I want to **handle unmatched routes gracefully**, so that **404 errors display a custom page and don't crash the app**.

---

## Context

The router supports a `notFound` callback option that renders custom 404 pages for unmatched routes:

```typescript
const router = createRouter({
  routes: [...],
  notFound: (ctx) => '<h1>Page not found: ' + ctx.path + '</h1>'
});
```

**Current State:**
- Feature implemented in `src/lib/router.ts:115-126`
- No unit tests for 404 fallback behavior
- E2E tests may cover basic 404, but edge cases untested

**Problem:** Without comprehensive tests:
- 404 page rendering may fail silently
- Cleanup functions not called for 404 routes
- Error states unhandled (notFound callback throws)

**Scope:**
1. Add unit tests for notFound callback invocation
2. Test 404 context object (path, matched, etc.)
3. Test cleanup functions for previous routes
4. Test error handling if notFound callback throws
5. Test fallback behavior when notFound not provided

**Out of Scope:**
- UI/styling of 404 pages (app-level responsibility)
- Analytics tracking of 404s (middleware, not router)

## Technical Notes

### Current 404 Implementation

**File:** `src/lib/router.ts:115-126`

```typescript
if (!chain.length) {
  if (opts.notFound) {
    const res = await opts.notFound(toContext);
    if (typeof res === 'function') currentCleanups = [res as () => void];
    else {
      currentCleanups = [];
      mountResult(outletEl, res as unknown as string | Node | DocumentFragment);
    }
  }
  currentContext = toContext;
  for (const h of afterHooks) h(toContext, fromContext);
  return;
}
```

### Test Scenarios

| Scenario | Setup | Expected Behavior | Test Type |
|----------|-------|-------------------|-----------|
| 404 with custom handler | `notFound: () => '<h1>Not Found</h1>'` | Handler called, HTML rendered | Unit |
| 404 context has correct path | Route `/nonexistent` not matched | `context.path = '/nonexistent'` | Unit |
| 404 with cleanup function | `notFound: () => () => console.log('cleanup')` | Cleanup function stored, invoked on next nav | Unit |
| 404 handler throws | `notFound: () => throw new Error(...)` | Error caught, logged (via onError handler) | Unit |
| 404 without handler | No `notFound` option | Navigation completes silently (no crash) | Unit |
| Previous cleanup called before 404 | Navigate route A → 404 route | Cleanup from A called before 404 rendered | Unit |

## Acceptance Criteria

- [ ] **AC1: notFound callback invoked**
  - When route doesn't match any defined route, `notFound` callback called
  - Callback receives correct context (path, params, query)
  - Unit test verifies callback invocation

- [ ] **AC2: 404 content rendered**
  - notFound callback returns string → rendered to outlet via `mountResult()`
  - notFound callback returns Node → appended to outlet
  - notFound callback returns DocumentFragment → appended to outlet
  - Content updates correctly on repeated 404 navigations

- [ ] **AC3: notFound cleanup functions**
  - If notFound callback returns a function, it's stored as cleanup
  - Cleanup invoked when navigating away from 404
  - Previous route's cleanup called before 404 rendered
  - Unit test verifies cleanup call order

- [ ] **AC4: 404 without handler**
  - If `notFound` not provided, navigation completes without error
  - Outlet not modified (no content mounted)
  - No console errors
  - Graceful fallback behavior

- [ ] **AC5: Error handling in notFound callback**
  - If `notFound` callback throws, error caught and logged
  - Error logged via router's `onError` handler (from S01-03)
  - App doesn't crash; navigation completes gracefully

- [ ] **AC6: 404 context object**
  - `context.path` = unmatched path
  - `context.params` = empty (no params on 404)
  - `context.query` = parsed query string
  - `context.matched` = empty array (no routes matched)

- [ ] **AC7: After hooks on 404**
  - After-navigation hooks called even for 404 routes
  - Hooks receive correct `to` and `from` contexts
  - Unit test verifies hook invocation

- [ ] **AC8: Coverage**
  - 404 code path achieves ≥90% coverage
  - All cleanup scenarios tested
  - Error path tested
