# Story S03-04 – Security enforcement: origin pinning (`http`) and HTTPS (`http_source`)

**Epic:** Security  
**Sprint:** 03  
**Points:** 2  
**Priority:** Must  
**Status:** pending  
**PRD refs:** NFR (Security), FR-03, FR-04

---

## User Story

As a **security-conscious maintainer**, I want the router to hard-enforce that `http` calls never leave the current origin and `http_source` calls always use HTTPS, so that **consumers cannot accidentally expose internal APIs cross-origin or send data over plain HTTP**.

---

## Context

Two security boundaries defined in the PRD:

1. **`http` (internal):** Final fetch URL must start with `window.location.origin`. A consumer cannot pass `http.url = 'https://evil.com/steal'` and have it work — the origin is always injected by the router, not the consumer.
2. **`http_source` (external):** Final fetch URL must start with `https://`. Plain `http://` is never used, even if the consumer's `url` somehow starts with `http://` after interpolation.

These invariants must be asserted in `resolveUrl` (in `fetcher.ts`) and covered by unit tests in S03-05.

---

## Acceptance Criteria

```gherkin
Given http.url = '/api/users/1'
When resolveUrl is called
Then the final URL starts with window.location.origin
And does NOT contain any external domain

Given http.url = '//evil.com/steal' (protocol-relative)
When resolveUrl is called
Then the url is treated as a path (origin is prepended), resulting in origin + '//evil.com/steal'
# Note: this produces an invalid URL — acceptable; fetch will fail with a network error,
# which surfaces in ctx.error. No special throw required.

Given http_source.url = 'api.example.com/data'
When resolveUrl is called
Then the final URL starts with 'https://'

Given http_source.url starts with 'http://' (consumer error)
When resolveUrl is called
Then resolveUrl strips the 'http://' prefix and prepends 'https://'
# Defensive normalization — never silently downgrade to HTTP

Given http_source.url starts with 'https://' (consumer already added scheme)
When resolveUrl is called
Then resolveUrl does NOT double-prepend, resulting in 'https://api.example.com/data'
```

---

## Technical Notes

**File to edit:** `src/lib/fetcher.ts` — `resolveUrl` function.

Defensive normalization for `http_source`:

```typescript
if (type === 'external') {
  // Strip any scheme the consumer may have accidentally included
  const stripped = config.url.replace(/^https?:\/\//i, '');
  const interpolated = interpolateParams(stripped, params);
  return `https://${interpolated}`;
}
```

Defensive check for `http` — the consumer cannot inject a foreign origin because the router always prepends `window.location.origin` and the `url` is treated as a **path**. No additional stripping needed; the origin is structurally enforced.

Add a **dev-mode warning** (not a throw) if `http.url` contains `://`:

```typescript
if (type === 'internal' && config.url.includes('://')) {
  console.warn(
    `[idae-router] http.url should be a path (e.g. "/api/users/:id"), not a full URL. ` +
    `Got: "${config.url}". Origin will be prepended regardless.`
  );
}
```
