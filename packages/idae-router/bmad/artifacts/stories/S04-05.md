---

## User Story

As a **library maintainer**, I want a comprehensive unit-test suite for Sprint 04 features so that **generics, cache behavior, and SWR router integration are validated at ≥ 90% coverage** before the sprint is considered shippable.

---

## Context

This story documents the full test scope for Sprint 04. Individual stories (S04-01, S04-03, S04-04) each include their own test obligations; this story defines the integration-level test contract and coverage gate.

Note: `src/lib/cache.test.ts` is created as part of S04-03. This story adds **integration-level** tests in `src/lib/router.test.ts` (extend existing file) and a type-level test in `src/lib/types.test-d.ts`.

---

## Acceptance Criteria

```gherkin
Given the full test suite runs via pnpm run test:unit
Then all tests pass with zero failures

Given pnpm run test:coverage
Then coverage for src/lib/cache.ts is >= 90%
And coverage for src/lib/router.ts is >= 75%
And coverage for src/lib/types.ts is >= 80% (type assertions)

Given Route<User> is declared in a type-only test file
When tsc --strict compiles it
Then ctx.data inside action is inferred as User | null (not unknown)

Given a router test with cache enabled and a mocked fetch
When navigating to a route twice
Then fetchRouteData is called once (cache hit on second navigation)

Given a router test with cache: false
When navigating to a route twice
Then fetchRouteData is called twice (no cache)

Given a router test with stale entry in cache
When navigating to the route
Then action is invoked immediately with stale data
And a background fetch is triggered
```
