# Story S03-05 – Unit tests for fetcher + router integration

**Epic:** Testing  
**Sprint:** 03  
**Points:** 2  
**Priority:** Must  
**Status:** pending  
**PRD refs:** All acceptance criteria from S03-01 → S03-04

---

## User Story

As a **maintainer**, I want comprehensive unit tests for the fetcher utility and its integration into the router, so that **regressions in data-fetching behaviour are caught automatically on every CI run**.

---

## Context

Tests follow the existing pattern in `src/lib/` (e.g., `matcher.test.ts`, `router.test.ts`). Native `fetch` must be mocked — Vitest's `vi.stubGlobal` or `vi.fn()` is the preferred approach. Tests run in the `jsdom` environment (see `vite.config.ts`).

---

## Acceptance Criteria

```gherkin
Given interpolateParams('/api/users/:id', { id: '42' })
Then returns '/api/users/42'

Given interpolateParams('/api/:a/:b', { a: 'x', b: 'y' })
Then returns '/api/x/y'

Given interpolateParams('/api/users/:id', {})
Then returns '/api/users/:id' (token left as-is when no match)

Given resolveUrl({ url: '/api/users/:id' }, 'internal', { id: '5' })
Then returns 'http://localhost/api/users/5'

Given resolveUrl({ url: 'api.example.com/users/:id' }, 'external', { id: '5' })
Then returns 'https://api.example.com/users/5'

Given resolveUrl({ url: 'http://evil.com/data' }, 'external', {})
Then returns 'https://evil.com/data' (http:// stripped, https:// prepended)

Given resolveUrl({ url: 'https://already.com/data' }, 'external', {})
Then returns 'https://already.com/data' (no double-prefix)

Given fetch is mocked to return { ok: true, json: () => ({ id: 1 }) }
When fetchRouteData({ http: { url: '/api/items/1' } }, {}) is called
Then result.data equals { id: 1 } and result.error is undefined

Given fetch is mocked to return { ok: false, status: 404 }
When fetchRouteData({ http: { url: '/api/items/1' } }, {}) is called
Then result.data is null and result.error.message equals 'HTTP 404'

Given fetch is mocked to reject with new Error('NetworkError')
When fetchRouteData({ http: { url: '/api/items/1' } }, {}) is called
Then result.data is null and result.error.message equals 'NetworkError'

Given a route with http defined and handleNavigation is called
When navigation completes
Then ctx.data in the action equals the mocked fetch response

Given a route with NO http or http_source
When navigation completes
Then ctx.data and ctx.error are both undefined (fetch is NOT called)

Given a route with both http and http_source defined
When fetchRouteData is called
Then console.warn is called once with a message containing 'http takes precedence'
And http is used (not http_source)
```

## Technical Notes

**New file:** `src/lib/fetcher.test.ts`

Test setup (add at top of test file):

```typescript
import { describe, it, expect, vi, beforeEach, afterEach } from 'vitest';
import { interpolateParams, resolveUrl, fetchRouteData } from './fetcher.js';

// Mock window.location.origin for jsdom
Object.defineProperty(window, 'location', {
  value: { origin: 'http://localhost' },
  writable: true
});
```

Mocking fetch:

```typescript
beforeEach(() => {
  vi.stubGlobal('fetch', vi.fn());
});
afterEach(() => {
  vi.unstubAllGlobals();
});
```

**Additions to `src/lib/router.test.ts`:**

Add 2–3 integration tests using a test route with `http` and a mocked `fetch`, verifying `ctx.data` is populated when `action` is invoked.
