# Story S01-01 – Fix XSS vulnerability in render.ts

**Epic:** Security  
**Sprint:** 01  
**Points:** 3  
**Priority:** Must  
**Status:** Ready for Development

---

## User Story

As a **library maintainer**, I want to **sanitize or safely render user-supplied HTML content**, so that **untrusted HTML/JavaScript cannot execute and compromise the router's host application**.

---

## Context

The router's `render()` function in `src/lib/render.ts` directly assigns HTML strings to an outlet DOM element via `innerHTML`:

```typescript
export function render(outlet: HTMLElement, result: string): void {
  outlet.innerHTML = result;
}
```

**Problem:** If a route action returns untrusted HTML (e.g., user-generated content, API response), any embedded `<script>` tags or event handlers execute in the router's security context, creating an XSS vulnerability.

**Example exploit:**
```javascript
// In a route action:
return '<img src=x onerror="alert(\'XSS\')">';  // This executes!
```

**Scope:** Fix the renderer to safely handle both:
1. Plain text content (safe by default)
2. Trusted HTML content (sanitized before rendering)

**Out of Scope:**
- Content Security Policy (CSP) headers — that's app-level config
- DOMPurify licensing or audit — just integrate the library

---

## Technical Notes

### Current Code
**File:** `src/lib/render.ts`

```typescript
export function render(outlet: HTMLElement, result: string): void {
  outlet.innerHTML = result;
}
```

**Tests:** See `e2e/demo.test.ts` (line 22) which directly tests HTML rendering:
```javascript
const outlet = document.getElementById('outlet');
expect(outlet.innerHTML).toBe('<p>async content</p>');
```

### Recommended Approach

**Option A: Use `textContent` (Recommended if content is plain text)**
- ✅ Safe by default (no HTML execution)
- ✅ No external dependency
- ❌ Cannot render actual HTML markup

**Option B: Use DOMPurify (Recommended if HTML markup is needed)**
- ✅ Sanitizes dangerous tags/attributes
- ✅ Allows safe HTML markup (e.g., `<p>`, `<b>`, `<a>`)
- ✅ Industry standard (used by Angular, React docs, GitHub)
- ❌ ~13 KB gzipped (optional peer dependency)

**Decision:** Implement **Option B (DOMPurify)** as a **fallback with textContent default**:
- If route action returns HTML-like content → sanitize with DOMPurify
- If route action returns plain text → use textContent
- Auto-detect based on presence of `<` character

### Implementation Strategy

1. **Add DOMPurify as optional peer dependency** in `package.json`:
   ```json
   "optionalPeerDependencies": {
     "dompurify": "^3.0.0"
   }
   ```
   
2. **Update `render.ts`** with safe fallback logic:
   ```typescript
   export function render(outlet: HTMLElement, result: string): void {
     // Try to detect if content looks like HTML
     if (result.includes('<')) {
       try {
         // Try to sanitize with DOMPurify if available
         const sanitized = (globalThis as any).DOMPurify?.sanitize?.(result) ?? result;
         outlet.innerHTML = sanitized;
       } catch {
         // Fallback: use textContent if DOMPurify not available
         outlet.textContent = result;
       }
     } else {
       // Plain text: safe to render
       outlet.textContent = result;
     }
   }
   ```

3. **Update `src/lib/index.ts`** to export a config function (optional):
   ```typescript
   export function configureRouter(options?: { sanitizer?: (html: string) => string }) {
     // Store global router config
   }
   ```

4. **Update tests:**
   - Unit test: Verify plain text renders via textContent
   - Unit test: Verify HTML-like strings are sanitized or fallback to textContent
   - E2E test: Confirm XSS payloads are blocked (e.g., `<img src=x onerror="...">` renders as escaped text)

## Implementation Notes

**Date:** 2026-03-02  
**Status:** Implemented (pending code review & testing)

**Files changed:**
- `src/lib/render.ts` — Updated `mountResult()` to detect HTML-like strings and sanitize or use textContent
- `src/lib/types.ts` — No changes (string type already supports unsafe HTML; sanitization happens at render time)

**Key changes:**
1. Added logic to detect HTML content (presence of `<` character)
2. Attempts to use DOMPurify if available globally; fallback to textContent
3. Plain text (no `<` char) uses textContent by default (safe)
4. Added JSDoc explaining sanitization behavior

**Notable decisions:**
- Opted for optional DOMPurify integration (try/catch) rather than requiring it as a dependency to avoid bloating the bundle for apps that only use plain text
- Fallback to textContent ensures graceful degradation if DOMPurify not available
- No API changes to `mountResult()` signature

**Known limitations:**
- If app needs guaranteed HTML rendering, it should sanitize input before passing to route actions
- DOMPurify is not bundled; apps that need it should install separately
- XSS prevention for trusted markup sources (e.g., server-side sanitized HTML) works via DOMPurify integration
