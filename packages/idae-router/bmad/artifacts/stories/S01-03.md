# Story S01-03 – Replace silenced catch() with proper error logging

**Epic:** Error Handling  
**Sprint:** 01  
**Points:** 2  
**Priority:** Must  
**Status:** Ready for Development

---

## User Story

As a **router user / developer**, I want to **see clear error messages when navigation fails**, so that **I can debug routing issues without silent failures**.

---

## Context

The router's `handleNavigation()` function silences errors during navigation:

**File:** `src/lib/router.ts` (lines 161, 172)

```typescript
// Line 161
handleNavigation(p).catch(() => {
  // Silently ignore errors
});

// Line 172
handleNavigation(p).catch(() => {
  // Silently ignore errors
});
```

**Problem:** When a route action fails (e.g., `beforeNavigate()` hook throws, async data fails to load), the error is swallowed, and developers see no feedback. This makes debugging navigation issues extremely difficult.

**Impact:**
- Developers cannot identify why routes fail to render
- Silent failures in production hide real bugs
- Impossible to set up global error handling or telemetry

**Scope:**
1. Replace `.catch(() => {})` with proper error logging
2. Add configurable error handler (optional)
3. Add tests verifying errors are logged
4. Update JSDoc to document error behavior

**Out of Scope:**
- Global error boundary UI component — that's app-level
- Error recovery strategies — just log and let app decide what to do

## Technical Notes

### Current Code

**File:** `src/lib/router.ts`

```typescript
function handleNavigation(path: string): Promise<void> {
  // ... route resolution, before hooks ...
  
  // Line 161 (incomplete navigation handling)
  handleNavigation(p).catch(() => {
    // Silently ignore errors
  });
  
  // Line 172 (similar pattern)
  handleNavigation(p).catch(() => {
    // Silently ignore errors
  });
}
```

### Recommended Approach

**Option A: Log to console (Simple)**
```typescript
handleNavigation(p).catch((error) => {
  console.error('Navigation failed:', error);
});
```

**Option B: Provide configurable error handler (Recommended)**
```typescript
interface RouterInstance {
  onError?: (error: Error, path: string) => void;
}

// In handleNavigation():
handleNavigation(p).catch((error) => {
  if (router.onError) {
    router.onError(error, p);
  } else {
    console.error('Navigation failed:', error);
  }
});
```

**Decision:** Implement **Option B (configurable handler with console.error fallback)**:
- Developers can optionally provide a custom error handler (e.g., for telemetry)
- Falls back to `console.error` if no handler provided
- Preserves current public API (no breaking changes)

### Implementation Strategy

1. **Add error handler to `RouterInstance` type** in `src/lib/types.ts`:
   ```typescript
   export interface RouterInstance {
     push: (path: string) => Promise<void>;
     before: (fn: BeforeNavigateFn) => void;
     onError?: (error: Error, path: string) => void;  // NEW
   }
   ```

2. **Update `router.ts` to call error handler**:
   ```typescript
   handleNavigation(p).catch((error) => {
     if (router.onError) {
       router.onError(error, p);
     } else {
       console.error(`[Router] Navigation to "${p}" failed:`, error);
     }
   });
   ```

3. **Add JSDoc**:
   ```typescript
   /**
    * Optional error handler invoked when navigation fails.
    * If not provided, errors are logged to console.error.
    */
   onError?: (error: Error, path: string) => void;
   ```

4. **Write tests**:
   - Unit test: Verify `console.error` called when `onError` not provided
   - Unit test: Verify custom `onError` handler called with (error, path)
   - Unit test: Verify error is passed with correct message

## Implementation Notes

**Date:** 2026-03-02  
**Status:** Implemented (pending code review & testing)

**Files changed:**
- `src/lib/types.ts` — Added `onError?: (error: Error, path: string) => void` to `RouterInstance` interface
- `src/lib/router.ts` — Replaced silenced `.catch(() => {})` with proper error logging on lines 161, 172, 207, 219

**Key changes:**
1. Added `onError` callback to `RouterInstance` type (optional, for custom error handling)
2. Created `handleError()` helper function that:
   - Calls `instance.onError()` if provided
   - Falls back to `console.error()` with formatted message
3. Updated all navigation error sites to call `handleError(error, path)` with proper error conversion
4. Error handler is stored in router instance variable (available during setimeout/event handlers)

**Notable decisions:**
- Made `onError` optional (backward compatible) — apps not providing it still work with console.error
- Used instance variable to avoid closure issues with `router.onError`
- Ensured all error types (Error objects, strings, etc.) are normalized to Error instances

**Known limitations:**
- None identified; fully backward compatible
