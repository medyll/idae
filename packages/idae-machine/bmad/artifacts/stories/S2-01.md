# Story S2-01 – Integration Testing Suite

**Epic:** Testing & Validation
**Sprint:** 2
**Points:** 5
**Priority:** **Must**
**Assignee:** Developer
**Status:** Ready

---

## User Story

As a **QA engineer**, I want to **create comprehensive end-to-end tests covering form submission, validation, data binding, and API integration**, so that **form workflows are fully validated and regressions are caught early**.

---

## Context

Sprint-1 delivered core functionality with 88 unit tests (100% pass rate). However, integration between components, validation pipeline, and data binding requires E2E coverage to ensure workflows function correctly end-to-end.

**Focus Areas:**
- Form submit with custom/async/cross-field validators
- Parent↔child data binding synchronization
- Error handling and recovery
- Reactive updates with Svelte 5 $effect pattern

---

## Acceptance Criteria

```gherkin
Feature: Form Submission
  Scenario: Submit form with valid data
    Given a form with required + optional fields
    When user fills all fields and clicks Submit
    Then form data is collected and validated successfully
    And validation errors = []
    And submit handler is called with clean data

  Scenario: Submit form with validation errors
    Given a form with custom validators registered
    When user enters invalid data (email format, password strength)
    Then validation fails before submit
    And error messages display inline
    And submit button remains disabled

Feature: Custom Validator Enforcement
  Scenario: Email format validation
    Given field 'email' with regex validator
    When user enters "invalid.email"
    Then validation fails with "Invalid email format" message

  Scenario: Password matching
    Given fields 'password' and 'passwordConfirm'
    When user enters mismatched passwords
    Then cross-field validation fails with "Passwords do not match"

Feature: Async Validator Handling
  Scenario: Username availability check
    Given field 'username' with async validator (API call)
    When user enters "takenUsername"
    Then async validator calls API and returns { available: false }
    And validation fails with "Username already taken"

  Scenario: Debounce async validation
    Given async validator with 300ms debounce
    When user types 5 characters in rapid succession
    Then API is called only once (after user stops typing)

Feature: Data Binding Synchronization
  Scenario: Parent updates child value
    Given parent component with formData = { name: "John" }
    When parent updates formData.name = "Jane"
    Then child input reflects new value immediately

  Scenario: Child updates parent value
    Given child input with bind:value={formData.name}
    When user types "New Value" in input
    Then parent formData.name updates immediately

  Scenario: Dual binding without infinite loop
    Given dual $effect pattern (parent→child, child→parent)
    When parent updates AND child updates simultaneously
    Then no infinite loop, final state is consistent

Feature: Error Handling
  Scenario: Network error during async validation
    Given async validator that calls API
    When API returns 500 error
    Then validation shows warning (not error)
    And form allows override/submission

  Scenario: Malformed field data
    Given field with strict type (number, date, etc)
    When parser receives malformed data
    Then type validation catches error
    And displays user-friendly error message
```

## Implementation Plan

### Phase 1: Setup Test Infrastructure (Day 1)
1. Create `src/lib/form/__tests__/` directory
2. Add integration test file: `CreateUpdate.integration.test.ts`
3. Setup test utilities: form mocking, field fixtures, validator stubs
4. Create test data factories for common form scenarios

### Phase 2: Form Submission Tests (Day 1-2)
1. Test valid form submission (all fields filled)
2. Test required field validation (missing fields)
3. Test custom validator enforcement (email, password, etc)
4. Test custom validator with multiple validators per field
5. Test form submit when validation fails (button disabled)

### Phase 3: Async Validator Tests (Day 2-3)
1. Test async validator registration
2. Test async validator execution on field blur
3. Test debounce behavior (300ms, only once)
4. Test network error handling (show warning, allow override)
5. Test multiple async validators on same field

### Phase 4: Cross-Field Validator Tests (Day 3)
1. Test cross-field validator registration
2. Test date range validation (endDate > startDate)
3. Test password matching validation
4. Test conditional requirements (field A requires field B)
5. Test error messaging for cross-field failures

### Phase 5: Data Binding Tests (Day 3-4)
1. Test parent→child updates (formData changes reflect in input)
2. Test child→parent updates (input changes update formData)
3. Test dual binding consistency (no infinite loops)
4. Test binding with nested form data
5. Test reactive updates with computed values

### Phase 6: Error Handling Tests (Day 4)
1. Test validation error display (inline messages)
2. Test error clearing on field update
3. Test network timeout handling
4. Test malformed parser data recovery
5. Test error state persistence

### Phase 7: Integration Scenarios (Day 4-5)
1. Full CRUD workflow with validation
2. Form reset with cleared validators
3. Multi-field dependent validation
4. Async + custom + cross-field validators together
5. Real-world form scenario (user registration)
