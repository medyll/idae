# S18-02 — groupBy prop on ExplorerList

**Sprint:** 18  
**Effort:** M  
**Priority:** high  
**Dependencies:** S18-01 (explorerUtils.ts must exist, SortBy type available)

## Context

ExplorerList renders a flat list. No grouping by field exists anywhere in the component tree.  
groupBy is needed for UX like "vehicles grouped by status" or "reservations grouped by date".

---

## Description

Add `groupBy?: string` (field name) prop to `ExplorerList`.  
When set, items are grouped into sections with a header per unique field value.  
Pagination remains global (across all groups combined), not per-group.

## Type + function to add in explorerUtils.ts

```ts
export function groupItems<T extends Record<string, unknown>>(
  items: T[],
  field: string
): Map<string, T[]> {
  const map = new Map<string, T[]>();
  for (const item of items) {
    const key = String(item[field] ?? '—');
    if (!map.has(key)) map.set(key, []);
    map.get(key)!.push(item);
  }
  return map;
}
```

Rules:
- `null`/`undefined` field value → group key `'—'` (em dash)
- Group insertion order = order of first occurrence in input array
- If `sortBy` also set, input is already sorted → groups maintain sorted order within

## ExplorerList change (script + template)

**Script:**
```diff
+ import { sortItems, groupItems } from './explorerUtils.js';

  interface ExplorerListProps {
     ...
+    groupBy?: string;
  }
  let {
+    groupBy,
     ...
  } = $props();

+ let groupedItems = $derived(groupBy ? groupItems(allItems, groupBy) : null);
```

**Template — replace the `<ul>` block:**
```svelte
{#if groupedItems}
  {#each groupedItems as [groupKey, groupItemList] (groupKey)}
    <section class="explorer-group">
      <h3 class="explorer-group-header">{groupKey}</h3>
      <ul class="explorer-list" role="list">
        {#each groupItemList as item (item[index])}
          <li class="explorer-item" role="button" tabindex="0"
              onclick={() => _onclick(item, item[index])}
              onkeydown={(e) => e.key === 'Enter' && _onclick(item, item[index])}>
            <CardFields collection={collection} data={item} mode="show" />
          </li>
        {/each}
      </ul>
    </section>
  {/each}
{:else}
  <!-- existing flat <ul> block unchanged -->
{/if}
```

Note: when `groupBy` is active, pagination slices `allItems` before grouping so page boundary
can split a group. This is acceptable — document in JSDoc. Alternative (paginate per group)
is out of scope for this story.

Actually: reconsider — pagination should still work. When `groupBy` set, `paginatedItems`
is still derived from `allItems` (sliced). Then `groupItems(paginatedItems, groupBy)` is used
for render, not `groupItems(allItems, groupBy)`.

**Correct derived:**
```ts
let groupedItems = $derived(groupBy ? groupItems(paginatedItems, groupBy) : null);
```

This ensures pagination + groupBy compose correctly.

## Acceptance criteria

1. `<ExplorerList collection="vehicle" groupBy="status">` → items render under `<h3>` headers per unique `status` value
2. Items within a group maintain their insertion order (or sort order if `sortBy` also passed)
3. Item with `null`/`undefined` `status` appears under group header `—` (em dash)
4. `groupBy` + `sortBy` together: items sorted first, then grouped → items within each group are in sort order
5. Omitting `groupBy` → flat render unchanged (no regression)
6. Pagination still functions when `groupBy` set (page boundary may split a group — acceptable)
7. `groupItems()` unit tests in `explorerSortGroup.test.ts`:
   - groups by string field
   - null/undefined → group '—'
   - preserves insertion order of groups
   - preserves item order within group
   - empty array returns empty Map
