# S18-03 — sortBy prop on ExplorerActions

**Sprint:** 18  
**Effort:** S  
**Priority:** medium  
**Dependencies:** S18-01 (explorerUtils.ts + sortItems must exist)

---

## Context

`ExplorerActions` is a flat menu list that takes a pre-fetched `data?: COL[]` array.  
It has no sort capability. When used as a side navigation (list of records), alphabetical
or date-sorted order is expected by users.

Unlike `ExplorerList`, `ExplorerActions` does NOT hit the store — it receives `data` as prop.
So `sortBy` applies to the passed array, not to a store query.

---

## Description

Add `sortBy?: SortBy` prop to `ExplorerActions`. Sort is applied to the `items` derived.

---

## Files to modify

| File | Change |
|------|--------|
| `src/lib/main-ui/explorer/ExplorerActions.svelte` | Import `SortBy` + `sortItems` from `explorerUtils.ts`; add `sortBy` prop; apply to `items` derived |

## ExplorerActions change

```diff
+ import type { SortBy } from './explorerUtils.js';
+ import { sortItems } from './explorerUtils.js';

  let { collection, target, data, style, where, onclick } = $props<{
     collection:   string;
     ...
+    sortBy?:      SortBy;
  }>();

- let items = $derived(data ?? []);
+ let rawItems = $derived(data ?? []);
+ let items    = $derived(sortBy ? sortItems(rawItems, sortBy) : rawItems);
```

No template change needed — `{#each items ...}` already correct.

---

## Acceptance criteria

1. `<ExplorerActions collection="vehicle" {data} sortBy={{ field: 'name', direction: 'asc' }}>` → list renders sorted by name asc
2. `direction: 'desc'` → reverse
3. Null field values last (same rule as `sortItems`)
4. No `sortBy` → identical to current behavior
5. Passing empty `data` array → no error, empty list

---

## Note on `where` prop

`ExplorerActions` currently declares `where` in its type but does NOT use it (no store access,  
`items` is just `data ?? []`). This is a pre-existing inconsistency. Do NOT fix it in this story.
