# S18-01 — sortBy prop on ExplorerList

## Context

`ExplorerList` currently derives `allItems` from the store and paginates. No sort prop exists.  
`ExplorerTable` has internal sort (click headers), but `ExplorerList` (card/line view) has none.

## Description

Add `sortBy?: { field: string; direction: 'asc' | 'desc' }` prop to `ExplorerList`.  
Sort logic must be extracted to a pure utility so it is testable without DOM.

## Type to add (in explorerUtils.ts)

```ts
export type SortBy = { field: string; direction: 'asc' | 'desc' };

export function sortItems<T extends Record<string, unknown>>(
  items: T[],
  sortBy: SortBy
): T[] {
  const { field, direction } = sortBy;
  return [...items].sort((a, b) => {
    const av = a[field] ?? null;
    const bv = b[field] ?? null;
    if (av === null && bv === null) return 0;
    if (av === null) return 1;   // nulls always last
    if (bv === null) return -1;  // nulls always last
    const cmp = av < bv ? -1 : av > bv ? 1 : 0;
    return direction === 'asc' ? cmp : -cmp;
  });
}
```

## Acceptance criteria

1. `<ExplorerList collection="vehicle" sortBy={{ field: 'name', direction: 'asc' }}>` → items render sorted by `name` ascending
2. `direction: 'desc'` → reverse order
3. Items with `null`/`undefined` for the sort field appear at the end, regardless of direction
4. Omitting `sortBy` → behavior identical to current (no regression on existing pagination tests)
5. `sortItems()` unit tests in `explorerSortGroup.test.ts`:
   - sorts strings asc/desc
   - sorts numbers asc/desc
   - nulls always last (both directions)
   - empty array returns empty array
   - returns new array (does not mutate input)
