# S28-04 — machine.fetchSchema() integration — drift after server schema change

**Sprint:** 28
**Effort:** S
**Priority:** high
**Dependencies:** S28-03

---

## Goal

Ensure `fetchSchema()` also triggers drift detection after receiving a new model from the server. Today fetchSchema() calls start() — that's sufficient if start() runs adaptIdbToSchema() first.

Verify the full flow:
```
fetchSchema(url)
  → GET /api/scheme → new model
  → this._model = newModel
  → this.start()          ← adaptIdbToSchema() runs here
    → drift detected if server schema changed
    → IDB upgraded
    → createQoolie with correct stores
  → sync pulls data from server into new stores
```

## Files

| File | Change |
|------|--------|
| `src/lib/main/machine.ts` | Verify fetchSchema calls await this.start() |

## Acceptance

- `fetchSchema()` becomes `async` (already is)
- calls `await this.start()` (not `this.start()`)
- stale-while-revalidate: if cached schema → start with cache, bg refresh may trigger 2nd start() + drift check
- No IDB data loss on same-schema reload (hash match → skip)
- Schema change from server → stores created/deleted correctly before sync begins
