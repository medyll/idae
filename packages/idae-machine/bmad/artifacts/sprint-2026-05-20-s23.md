# Sprint 23 — Refactor structurel : data-ui + shell

**Date:** 2026-05-20  
**Goal:** Séparer clairement les composants data-aware (`src/lib/data-ui/`) des composants shell/composition (`src/lib/shell/`), restaurer `DataList` et créer `SchemeList`, câbler `machine.loadIn()` partout.

**Références:** NAVIGATION.md §11, diagnostic session 2026-05-20

---

## Contexte et motivation

Architecture actuelle `src/lib/main-ui/` mélange :
- Composants **data-aware** (machine.store + machine.logic) : DataList, DataForm, DataFields, DataFk, DataRfk
- Composants **shell** (composition + layout) : ExplorerList, PaneLeft, CardCreate, CardEdit
- Composants **atomiques** : field/, input/, fragments/

Problèmes constatés :
1. `ExplorerList` est monolithique (data + UI + pagination + sort fusionnés)
2. `PaneLeft` duplique la logique collection-listing inline
3. `hydrate()` direct dans ExplorerList + ExplorerActions bypasse `machine.loadIn()`
4. Pas de `SchemeList` réutilisable (liste de schémas/collections)
5. `main-ui/` mauvais nom — ne reflète pas la nature des composants

## S23-02 — Restaurer `DataList` dans `data-ui/data/`

**Source :** Restaurer depuis git `8ced1ba9~1:packages/idae-machine/src/lib/main-ui/lists/DataList.svelte` et adapter à l'API actuelle (qoolie au lieu d'idae-idbql, machine.store au lieu d'idbqlState).

**Rôle :** Composant data provider pur. Fetche les records, expose slot pour renderer. Réutilisable par ExplorerList, ExplorerActions, PaneRecents.

```svelte
  pageSize={20}
  let:items
  let:index
  let:fieldValues
  let:pagination
>
  <!-- renderer ici -->
  {#each items as item}
    <slot {item} {index} {fieldValues} />
  {/each}
</DataList>
```

**Acceptance criteria :**
- `src/lib/data-ui/data/DataList.svelte` créé
- Fetche depuis `machine.store[collection].getAll()` ou `.where(where)`
- Expose `items`, `index`, `fieldValues`, `pagination` via slot ou snippets
- Supporte `sortBy` (SortBy | SortBy[]), `groupBy`, `pageSize`
- Réactif aux changements de `signals.dataVersion`
- defaultSort depuis `machine.logic.collection(c).defaultSort`
- Aucun HTML de rendu inliné — data provider pur
- Gestion d'erreur si collection inconnue

**Tests :** `dataList.test.ts` — getAll, where, sort, page, erreur collection inconnue

## S23-05 — Créer `SchemeList` + `SchemeItem` dans `data-ui/scheme/`

**Rôle :** Liste les collections depuis `machine.logic.collections()`. Réutilisable par PaneLeft, navigation, tout composant qui liste des schemas.

**API cible :**
```svelte
<SchemeList
  filter={query}
  onSelect={(collection) => ...}
  activeCollection="vehicle"
  let:groups
>
  <!-- renderer par défaut groupé par type -->
</SchemeList>
```

**Logique interne :**
- `machine.logic.collections()` → grouper par type (isType/isGroup/standard)
- Filtrer par `machine.rights.checkAccess(col, 'R')`
- Filtrer par `filter` query string
- Expose `groups: Array<{ type, collections: Array<{code, name}> }>`

**SchemeItem :** Rendu d'un item de collection (nom, icône selon type). Utilisé par PaneCollectionGroup.

**Acceptance criteria :**
- `src/lib/data-ui/scheme/SchemeList.svelte` créé
- `src/lib/data-ui/scheme/SchemeItem.svelte` créé
- Groupement type (standard/type/group) fonctionnel
- Filtre sur query string
- RBAC via machine.rights.checkAccess
