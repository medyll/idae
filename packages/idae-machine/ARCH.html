Carte des rôles par couche

Machine core

src/lib/main/machine.ts

Orchestrateur du runtime : boot, résolution du schéma, création du store IDB (qoolie), wiring socket/sync, injection router/framer, chargement des droits, API publique machine.*. Singleton, ne jamais importer les internes.

boot() logic store() collection() framer

Cerveau du schéma

machineDb.ts + machineParserForge.ts

Couche pure d'introspection. Parse les règles de champ, expose les collections comme MachineScheme, calcule vues, types, labels, defaults, validation. Pas d'I/O, déterministe.

pure parser views field metadata

Owner du meta-schema

server/src/MachineServer.ts

Possède le meta-schema autoritaire en Mongo. Relit appscheme_*, reconstruit un MachineModel, sert /api/scheme, déploie les modèles, démarre routes (auth/data/files/mail) et socket IO.

/api/scheme deployModel() socket routes

Pipeline seed / deploy

server/src/bootstrap/*.ts

Transforme un MachineModel en lignes engine : registries, collections, fields, relations, views, puis seed des users et des données métier. Pont entre le code déclaratif et les meta-documents Mongo.

deployModel() seedUsers() seedBusinessData() bootstrap-demo.ts

Runtime storage / sync

qoolie + schema cache + IDB drift adapter

Une fois booté, le client vit sur IndexedDB. Schéma caché, drift détecté, stores migrés. sync + socket repoussent l'état server dans la data locale réactive. Reads = store · CRUD = collection (ne jamais mélanger).

IndexedDB schema cache drift sync

Shell + UI générique

shell/* + data-ui/*

Le shell donne les zones et points de montage (Frame dynamique). La data-ui consomme la metadata du schéma et rend records, forms, lists, relations FK/RFK et navigation sans code par collection. Atomes de champ show+edit unifiés sous field/snippets/ (dispatch par mode) ; toolbars externalisées (FicheToolbar · ButtonAction · Sort/Group, médiateur useMachinePrefs).

Frame.svelte TemplateShell DataList field/snippets toolbars composables

Frame zones & sizing

Zones (data-target-zone)

main seule zone DOM réelle — lists, forms, frames loadInDialog() overlays / dialogs flottants (content-keyed) main.panel / .modal / .window abandonnés — jamais montés dans le DOM

⚠ main est la seule zone avec une cible DOM. Tout le flottant (dialogs, panneaux, fenêtres) passe par machine.framer.loadInDialog(...), pas par une zone nommée. Les zones main.panel/modal/window furent documentées mais jamais implémentées — nommage abandonné 2026-06-10.

createHost(target, { fill })

fill: true (défaut) → position:absolute; inset:0. Correct pour une zone dimensionnée (main, panel). Utilisé par shell/Frame.svelte.

fill: false → position:relative (flux normal). Requis pour les hôtes content-driven (Dialog flottant auto-dimensionné).

⚠ Trap : dialog hôte avec fill:true → n'affiche que la 1ère ligne (hôte collapse, contenu overflow invisible). Vérifier ça en premier.

Règles runtime clés

Boot first

machine.init() puis await machine.boot() avant store/logic/shell.

Server possède le core schema

La baseline est déployée en Mongo puis servie. Le client n'importe pas le core comme vérité live.

Le parser reste pur

MachineParserForge ne parse que règles + metadata. Pas d'I/O, pas d'effets de bord.

Reads vs CRUD

machine.store (réactif) pour lire, machine.collection (impératif) pour écrire. Mélanger = qoolie dual-bus → reads vides.

Framer = pont UI

Nav résolue en zones + composants via MachineFrameManager + componentRegistry, pas de mounts hardcodés.

Seed ≠ runtime

Bootstrap écrit le meta-schema en Mongo. Le boot relit puis crée le storage local opérationnel.

Custom tag = display explicite

Chaque tag custom déclare son display CSS. Manquant = inline = layout cassé.

Runes Svelte 5 only

$state · $derived · $effect · $props · $bindable. Pas de $:, onMount, export let.

Jamais de self-import

Dans src/, jamais from '@medyll/idae-machine' — toujours $lib ou relatif. Le self-import résout vers dist/ et fait type-checker le JS compilé (erreurs fantômes).

Modèle mental en une ligne

Server/bootstrap fabrique et stocke le schéma machine-readable, machine core boote un runtime local réactif à partir de lui, et shell/data-ui rendent des interfaces génériques en consommant ce schéma parsé au lieu de hardcoder des écrans métier.
