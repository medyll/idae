# S1-05 — Créer `batch.ts` + intégrer dans `proxy.ts`
**Effort:** M | **Sprint:** 1 | **Dépend de:** S1-04

## Contexte
Actuellement, chaque mutation déclenche immédiatement un événement `stator:change`. Il faut ajouter un mécanisme de batch pour grouper plusieurs mutations en une seule notification. Ce mécanisme sera global (module-level) — suffisant pour l'usage synchrone.

## Fichier à CRÉER
`src/lib/stator/batch.ts`

## Contenu EXACT du fichier `batch.ts` à créer

```typescript
/**
 * Global batch context.
 * When active, notifications are queued instead of fired immediately.
 * At the end of batch(), all queued notifications are flushed once.
 */
const batchContext = {
  active: false,
  pending: new Set<() => void>(),
};

/**
 * Returns true if a batch is currently in progress.
 */
export function isBatching(): boolean {
  return batchContext.active;
}

/**
 * Queue a notification to be fired after the current batch ends.
 * If no batch is active, the notification is fired immediately.
 *
 * @param fn - the notification callback to queue or fire
 */
export function queueNotification(fn: () => void): void {
  if (batchContext.active) {
    batchContext.pending.add(fn);
  } else {
    fn();
  }
}

/**
 * Execute fn() and defer all state-change notifications until fn() completes.
 * Only one notification per queued callback will fire, even if the same
 * callback was added multiple times.
 *
 * @param fn - the function that performs multiple mutations
 *
 * @example
 * import { batch } from '@medyll/idae-stator';
 * batch(() => {
 *   state.value.x = 1;
 *   state.value.y = 2;
 * }); // → only one 'stator:change' event fired
 */
export function batch(fn: () => void): void {
  batchContext.active = true;
  try {
    fn();
  } finally {
    batchContext.active = false;
    // Flush all pending notifications
    for (const notify of batchContext.pending) {
      notify();
    }
    batchContext.pending.clear();
  }
}
```

## Pièges à éviter
- `batchContext.pending` est un `Set` : si la même fonction est ajoutée plusieurs fois, elle n'est exécutée qu'une fois — c'est voulu
- Le `try/finally` dans `batch()` est obligatoire : sans lui, une exception dans `fn()` laisserait `batchContext.active = true` pour toujours
- Ne pas appeler `ctx.notify()` directement dans `proxy.ts` — toujours passer par `queueNotification`
