# S1-07 — Ajouter `subscribe()` sur `AugmentedState`
**Effort:** S | **Sprint:** 1 | **Dépend de:** S1-06

## Contexte
`subscribe()` est implémenté dans S1-06 en tant que méthode de l'objet `augmented`. Cette story vérifie que l'implémentation est correcte, que les types sont bons, et documente le comportement attendu de façon exhaustive.

## Vérification du code produit par S1-06

La méthode `subscribe` doit avoir exactement cette signature dans `AugmentedState<T>` (défini dans `types.ts`) :

```typescript
subscribe: (listener: StateChangeHandler<T>) => () => void;
```

Et cette implémentation dans `stator()` :

```typescript
subscribe(listener: StateChangeHandler<T>): () => void {
  const handler = (e: Event) => {
    const { oldValue, newValue } = (e as CustomEvent).detail;
    listener(newValue, oldValue);  // ← ordre : newValue PUIS oldValue
  };
  eventTarget.addEventListener('stator:change', handler);
  return () => eventTarget.removeEventListener('stator:change', handler);
},
```

**Important :** `StateChangeHandler<T>` est défini comme `(oldValue: T, newValue: T) => void`. Mais dans l'événement `stator:change`, le `detail` est `{ oldValue, newValue }`. Pour respecter la signature du handler, l'appel est `listener(newValue, oldValue)` — car `subscribe` passe `newValue` en premier.

Attends — c'est ambiguë. Vérifions la convention :
- `onchange` est appelé via `onchangeListener = (e) => onchange?.(e.detail.oldValue, e.detail.newValue)` → `onchange(oldValue, newValue)`
- Pour cohérence, `subscribe` doit aussi appeler `listener(oldValue, newValue)`

**Correction :** l'implémentation correcte est :
```typescript
subscribe(listener: StateChangeHandler<T>): () => void {
  const handler = (e: Event) => {
    const { oldValue, newValue } = (e as CustomEvent).detail;
    listener(oldValue, newValue);  // ← cohérent avec onchange
  };
  eventTarget.addEventListener('stator:change', handler);
  return () => eventTarget.removeEventListener('stator:change', handler);
},
```

## Ce que cette story fait concrètement

1. **Vérifier** que S1-06 a bien implémenté `subscribe` avec l'ordre `listener(oldValue, newValue)`
2. **Corriger** si l'ordre est inversé
3. **Vérifier** que `types.ts` contient bien la signature de `subscribe`

## Cas d'usage à valider manuellement

```typescript
// Usage 1 : unsubscribe
const state = stator({ count: 0 });
const unsub = state.subscribe((oldVal, newVal) => {
  console.log('changed:', oldVal, '->', newVal);
});
state.value.count = 1; // → logs "changed: {count:0} -> {count:1}"
unsub();
state.value.count = 2; // → rien (unsubscribed)

// Usage 2 : plusieurs subscribers indépendants
const s1 = state.subscribe(() => console.log('s1'));
const s2 = state.subscribe(() => console.log('s2'));
state.value.count = 3; // → logs "s1" et "s2"
s1();
state.value.count = 4; // → logs "s2" seulement

// Usage 3 : compatibilité avec Svelte 5 $effect
// $effect(() => {
//   const unsub = state.subscribe((o, n) => updateUI(n));
//   return unsub; // Svelte appelle le retour au cleanup
// });
```

## Acceptance Criteria
- [ ] `state.subscribe(fn)` retourne une fonction `() => void`
- [ ] La fonction retournée, appelée, stoppe les notifications vers `fn`
- [ ] `fn` reçoit `(oldValue, newValue)` dans cet ordre (cohérent avec `onchange`)
- [ ] Plusieurs `subscribe` simultanés fonctionnent indépendamment
- [ ] Unsubscribe d'un subscriber n'affecte pas les autres
- [ ] `npm test` passe (les tests S1-08 valideront en détail)

## Pièges à éviter
- Confondre l'ordre `(oldValue, newValue)` vs `(newValue, oldValue)` — utiliser `onchange` comme référence
- Ne pas `removeEventListener` avec une fonction anonyme différente de celle passée à `addEventListener` — stocker la référence dans `handler`
